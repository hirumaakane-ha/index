Night 2 - Mon, Jul 9th, ’18:

Copping a Domain and SSL…

  • My website adventure continued after the sushi fes that I consumed the afternoon before. I had to choose between Apache2 and nginx as the HTTP web server for my Ubuntu server machine. Somehow it didn’t take me long to reach a decision for Apache. Maybe it was because Apache sounded phonetically catchier and rememberable compared to nginx(?)
  • 2:30AM, I finally finished the initial setup for my server with Apache installed. Then, I needed to get a certificate for secured connection on my website. Now, at the time of writing this post, I still don’t know the deep implications that the certification process had to do, maybe I’ll figure it out in the future. I used Letsencrypt package to get certified. However, little did I know nor did the tutorial specify, Letsencrypt needed a challenge program or, in this case, Certbot to do whatever it needed to do for my website certification. I found this out by Googling. So, I installed Certbot from probably a non-canonical repository as far as I could remember. And, the certification process worked like a charm.
  • After I got certified, I realized it was around 4AM already but I still pushed on. In retrospect, I did the initial setup of a web server by installing (A)pache. But I did not completely setup the LAMP stack. So, I reviewed on my overlook and drew myself a lesson: don’t always diverge from the original tutorial because it might mess up my progress flow. Anyway, I traced back to where I left in the complete tutorial and managed to complete my LAMP stack for the server. However, everything did not go so smoothly with mysql hogging on my ass for centuries with authentication failures. But, by following some godly tutorials that I found on Google, I barely successfully set up the M in LAMP. For the (P), I have not looked into it further since my brain was already a mess at the time.
  • Karma is a bit fast lately as to balance out the very huge setback with mysql, it gave me the blissful and smooth-sailing domain name acquisition and DNS redirecting. I bought my domain name, which was this website, hoanganhramen.xyz for a meagre $20 barebone option that came WHOIS lookup prevention on most of the information from internetbs.com.
  • The registration process for the domain name was pretty straight forward as it asked for my contact information. It was like registering your ownership for a car, not that I had done such process before but you could think of it that way. Now. The DNS, short for Domain Name System, takes the human recognizable domain names (i.e., hoanganhramen.xyz) and converts it to computer language which is frequently referred to as IP address (i.e., 35.170.187.95)
  • After I had taken ownership of my dazzlingly brand new domain name, I must then associate the name with the public-facing IP address of my EC2 web instance. So, I did some digging around the AWS Console and found out about Route 53 included in the AWS. I opened the Route 53 and created a new hosted zone. By copying the nameservers offered by Route 53 configurations, I input those into the domain name settings on my registrar’s (internetbs.com) management. I then needed to wait for about 2-3 hours for the changes to propagate throughout the internet and services and whatever it was.
  • AND bam, I jumped for joy as my domain name was then in working conditions.
